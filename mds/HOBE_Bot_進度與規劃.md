## 三、接下來的路徑

### 階段一：流程設計（需要你的決策，不需寫程式）

把家長端、老師端的完整對話流程一步一步定下來。卡關點是上面那三個待決定問題。
家長請假流程目前已大致成形：

> 家長傳訊息 → Bot 判斷意圖 → 撈出孩子課表 → 家長選哪一堂 →
> 確認 → 標記請假寫入行事曆 → 進入補課媒合 → 媒合完成 → 雙方確認

### 階段二：寫程式

1. **Bot 加上「對話狀態記憶」** — 目前 Bot 收一句回一句、無記憶。補課協調需要它記得每個人進行到哪一步。
2. **串接 Google Sheets** — 讀家長對應表，用 LINE userId 找到學生。
3. **串接 Google Calendar** — 讀 6 個行事曆、寫入補課/調課行事曆。
4. **設定老師識別** — 把老師的 LINE userId 填進 TEACHER_IDS。
5. **接上意圖判斷與補課媒合邏輯**。
6. **補課排定後的訊息推播** — 媒合完成後自動推播確認訊息。流程：先詢問家長該時段是否可以，確認後再通知老師。訊息格式需考慮 LINE 的顯示方式（純文字或 Flex Message）。對應排程系統 todos 第三項「生成可傳給家長與老師的訊息」，會先做訊息生成，之後再接 LINE 推播。

### 階段三：測試

自己當家長、找人當老師，完整跑一次補課流程，抓問題。

### 階段四：正式上線

1. 等 HOBE 正式官方帳號的最高管理員開通 Messaging API。
2. 把測試帳號的 token 換成正式帳號的。
3. 老師、家長逐步加入官方帳號。

## 四、這個系統可以做得更好的地方

### 1. 家長對應靠人工填，110 筆很費工且易錯
目前要人工把每個家長的 LINE userId 對到學生。建議之後做 **QR code 自動綁定**：每個學生一個專屬 QR code，家長掃碼加好友，Bot 自動完成對應，零人工。

### 2. 同名學生的處理很脆弱
資料裡有多個「子晴」，靠名字加括號（如「（國小）子晴」）區分。這種命名混在正式資料裡，長期會出問題。建議：所有比對一律用學生 id（唯一），名字欄位保持乾淨，不要塞括號或年級。

### 3. Bot 目前無狀態記憶
收一句回一句，沒辦法處理多輪對話。補課協調一定需要狀態管理，這是階段二的第一件事。

### 4. 安全性：Service Account 金鑰需注意
Firebase Service Account 的私鑰先前曾在對話中傳遞。雖然是為了讀資料，但最佳做法是**之後到 Firebase Console 重新產生一把新金鑰**，舊的作廢，降低外洩風險。

### 5. ~~排程系統還在 localhost，行政其實沒在用~~（已解）
已於 2026-05-14 部署 GitHub Pages、2026-06-11 打包成 PWA，手機可安裝成 App，行政端隨時可用。

### 6. 學費自動化值得平行規劃
老闆希望先做補課，但學費流程其實更單純（每月定額、通知、收確認、催繳），自動化的投入產出比很明確。可以先把學費流程的規劃寫下來，補課做完接著做。

### 7. Bot 需要容錯與 fallback
家長一定會傳 Bot 看不懂的訊息。需要設計：聽不懂時怎麼回、什麼情況轉真人、出錯時怎麼通知你。否則家長體驗會卡。
