## 一、目前的架構長什麼樣

### 整體拓樸

- 1 個入口 HTML（`補習班排程系統.html`）+ 1 隻 JS（`app.js`）+ 1 份 CSS（`style.css`）
- 沒有打包工具、沒有 module、沒有測試
- 所有函式掛在 `window` 全域，由約 47 個 inline `onclick=` 觸發
- 第三方依賴皆走 CDN：Firebase Auth/Firestore、Google Identity Services、GAPI Calendar v3

### 功能分區（依 app.js 的 `// ──` 區塊）

| 行數 | 區塊 | 角色 |
|---|---|---|
| 1–50 | 常數 / 全域狀態（25+ 個 `let`） | 設定中心 |
| 50–200 | OAuth、Firestore、navigation | 啟動與面板切換 |
| 209–306 | `parseEv` / `buildTitle` | **核心解析（純函式）** |
| 308–520 | 今日視圖載入 + 渲染 + 教室時間軸 | 課程頁主畫面 |
| 600–725 | 本週視圖 | 週覽 |
| 820–1032 | 請假面板（toggle / select / preview / confirm / cancel） | 請假流程 |
| 1035–1247 | 待補課/調課清單 + matchMap | 補課總覽 |
| 1288–1635 | 補課時段選擇器（日 → 時段 → 教室 → 確認） | 安排補課 |
| 1640–1930 | 學生掃描、CRUD、彈窗 | 學生管理 |
| 1938–2173 | 新增課程 + 學生編輯 | 表單流程 |

### 資料流

```
Google Calendar ──fetch──▶ dayEvents / weekEvents / makeupList ──▶ render*  ──innerHTML──▶ DOM
                                                                       │
Firestore (sharedData/main) ──▶ driveData ──▶ getStudentList / getMakeupScheduled
                                                                       │
補課/調課行事曆掃描 ──▶ makeupMatchMap（id ↔ 已安排記錄）
```

## 二、會痛的地方（依建議處理順位排列）

### 1. 資安：Firestore 任何人都能讀寫 ⚠

目前用 `signInAnonymously()` 取得 Firebase Auth，整份 `sharedData/main`（全部學生姓名 + 補課排程）只要拿到 Firebase web config（寫在原始碼）就能讀寫。**等於學生名單對外公開。**

→ 處理：加 Firestore Security Rules 限定 email allowlist，或改成必須以特定 Google 帳號登入才能存取。

### 2. 沒有 EventStore，到處 `[...dayEvents,...weekEvents].find()`

這個模式在「選課程開 modal」「確認請假」「取消請假」「確認調課」「開 slot picker」等十幾處重複。導致：

- 同一個 event 在 dayEvents、weekEvents 可能各存一份，資料更新後容易不同步
- 新增資料來源（如 `makeupList`）就要全檔搜尋補進去

→ 處理：抽出單一 `eventCache.byId(id)`；順帶把重複的 `calendar.events.list` 加一層快取。

### 3. app.js 已 2173 行，但拆分線早就畫好了

程式碼本身已用 `// ──` 把功能分區，直接照線拆成 6 個檔案就能立刻降低編輯負擔：

- `init.js`（認證 / Firestore / 導覽）
- `parse.js`（事件解析 — **同時加單元測試**）
- `today.js` + `week.js`
- `absence.js`（請假面板）
- `makeup.js`（清單 + slot picker）
- `students.js`（管理 + 掃描 + 新增課程）

維持單一 `<script>`（concat）或改 `type="module"` 兩種都可行；初步建議 concat 保留現狀，行為零變更。

### 4. UI 完全靠 innerHTML + 全域 onclick（XSS 風險 + 維護痛）

40 個 innerHTML 寫入 × 47 個 inline onclick。雖然有 `esc()` 函式，但容易漏寫（學生姓名、課程標題、備註全來自 Google Calendar 事件，等同使用者輸入）。風險：

- 若有人在 Calendar 事件標題塞 `<script>` 或屬性注入字串，就要 100% 仰賴 esc() 從未漏寫
- 所有 handler 必須是 global → 永遠無法 module 化

→ 處理：長期目標。短期可先把「面板狀態」從 DOM class 改成 state 物件（`absState`、`slotPicker` 已是此方向）。

### 5. `renderMakeup`（120 行）混了三件事

篩選 / 統計 / HTML 拼接全在同一個函式裡。改一個 badge 顏色要先讀完整段。

→ 處理：抽 `computeMakeupStats(period)` 與 `mkCardHtml(e, kind)`，render 只負責組裝。

### 6. 重複的 Calendar API 呼叫沒有快取

切換週、切換日、開 slot picker、loadMakeup 各自打 `events.list`。每次切「前一天」就把整套今日 + 整週重撈。

→ 處理：加 `fetchRange(start, end)` 一層 LRU，或讓 `loadWeek` 結果順帶 derive 今日。

### 7. 雙寫 fallback（localStorage ↔ Firestore）是死路

`loadFromFirestore` 失敗才退 localStorage，但 `saveStudentList` 只寫 driveData → 排程後寫 Firestore，離線時其實沒人接。學生資料雲端化已完成，這段 fallback 是歷史包袱可清。

### 8. 全域狀態 25 個 `let` 散在檔案各處

如 `mkOpenId` 在第 1859 行才宣告。讀程式碼時要前後跳。拆檔時順手集中到各模組頂端即可。

## 三、相對健康的部分（不用動）

- **`parseEv()`**：唯一一個純函式核心，邏輯清楚、輸入輸出明確，**最適合先寫單元測試**
- **CSS 用 CSS variables**（`--bg`、`--ac`…）做主題色，色票統一
- **Firestore debounce save**（`scheduleDriveSave` 1.5s）寫法乾淨
- **`slotPicker` 物件 + `renderSpBody` 重渲染模式**，是全檔最接近「state → view」的地方，可以當未來其他面板的範本

---

## 四、建議的下一步順序

1. **加 Firestore Security Rules**（資安、半小時可完成）
2. **拆 app.js**（純機械工，零行為變更，馬上回本）
3. **為 `parseEv` 寫單元測試**（之後改解析規則才不用每次手測）
4. **抽 EventStore + Calendar API 快取**（降低後續所有 feature 的維護成本）
5. **逐步把面板邏輯改成 state-driven**（長期目標，從新功能開始用）
