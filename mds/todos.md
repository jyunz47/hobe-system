## 開發路線

> 下次回來從第一個 ⏳ 開始。
> **北極星**（每步對齊的終點）：核心操作（管學生／課程／學費）全在系統內完成、不必碰 Google Calendar；最終讓 Calendar 降為系統自動產生的投影。完整說明見 [`系統地圖.md`](系統地圖.md)「北極星」段。**目前約四成**：歸屬與學費已進系統，缺「課程實體」（老師／時段／教室）與雙向同步。不翻桌、走漸進路線。

> **過渡期定位（2026-06-16 敲定）**：現階段是「**開發者讓系統接軌真實情況**」的過程，**不是**讓行政改用系統。行政維持現狀（iCloud 行事曆）不動，等系統成熟、實戰夠穩才整批切換（含 Mac 行事曆設定／全員 onboarding，那時才碰）。這之前由開發者：對齊行事曆 → 輸入真實學費資料 → 把學費/點名做到能測。
> **首要：對齊行事曆 + 學費系統**；點名低風險、可平行穿插。

- [ ] ⏳ **對齊行事曆（把真實資料灌進系統，當學費/點名的測試底）**
  - 為什麼：學費/點名都讀系統（= Google 行事曆）；iCloud 上行政新增的學生/課/調度系統看不到 → 不對齊，學費就是在算假資料
  - 步驟：iCloud 匯出 .ics →（⚠️ Google 匯入**不去重**，先用一本測試行事曆試、確認備註格式有活下來）→ 匯入對應 Google 六本 → 系統登入驗證 → **課表對帳**把 enrollment 盤乾淨（查無此人→建檔登記、修課差異→補登）→ 輸入價目表/個人覆蓋價
  - ⚠️ 這是**快照、非永久收斂**：行政仍在寫 iCloud，真正 cutover（行政切 Google）那天要再對齊一次補空窗
  - **Phase 0 盤點（2026-06-16 已做，匯入前必看）**：
    - 系統靠 `calendarList` **完全比對名稱**抓六本（init.js:240）→ 登入帳號的日曆清單裡必須有名稱**完全一致**的：一般課程／補課／調課／試聽／練習課／加課。差一字就抓不到 → 匯入務必匯到名稱相符的那本
    - 學生統計/課表對帳只掃 **一般課程／練習課／加課** 三本（students.js `SCAN_CALS`）；補課/調課/試聽不進掃描
    - 備註格式（系統解析依據）見 [`功能說明.md`](功能說明.md)「課程備註格式」「請假標題格式」：第一行＝[教室 ]老師、第二行起＝學生（頓號分隔，練習課可「科目：學生」）、課型靠標題、請假/曠課/調課靠標題【】。匯入後抽幾筆確認 DESCRIPTION 有保留此結構
  - 預估：操作為主，工程支援約半天

- [ ] ⏳ **學費系統：計算 + 學費條 + 收據**（首要）
  - 為什麼：要結算學費、給家長收據
  - 子項：
    - 學費頁（左側選單新增）：月度視角，列出本期所有學生 × 應繳金額、勾選誰已收
    - 學生 modal 加「繳費紀錄」分頁：個別查歷史
    - 學費條 PDF 產生器
    - 收據 PDF 產生器
  - 前置：
    - ~~收費公式~~（2026-06-12 已定：**每堂單價 × 堂數**）
    - ~~單價歸屬~~（2026-06-12 已定：**課程預設價 + 個人覆蓋**，已實作於 coursePrices + enrollment.price）
    - **對齊行事曆完成 + enrollment 盤乾淨 + 價目表輸入完**（上一項）
    - **學費條 / 收據格式**：有現成範本嗎？或要從零畫？
  - ⚠️ **堂數來源**：點名未做前，過渡期「實際堂數」從「**排定課程數 −（不補課的請假/曠課）**」推算（行事曆衍生）；點名做好後改用實際出席精修。學費第一版的堂數是推算的、非點到的
  - 預估：4~5 個工作段

- [ ] ⏳ **每日營運：點名 + 成績登記 + 高中數學作業**（低風險、可平行穿插，不擋學費）
  - 為什麼：每堂課都要點名 → 點名紀錄是學費「實際堂數」更精準的事實來源（取代上面的推算）；成績與作業是每日營運需求
  - 已定規格（2026-06-12）：
    - 課程設定加「需登記成績」開關（每門課手動設定，新課預設只點名）；今日課程分兩區：成績課在上、只點名的課在下（區內維持時間排序）
    - 點名只記「到/未到」；已請假/曠課的學生自動帶入顯示、不重複操作；臨時沒到的從點名面板跳既有請假/曠課流程
    - 成績：每堂每生可登**多筆**（標籤＋分數），涵蓋課前考、練習課重寫第二份考卷的情境
    - 高中作業：每天一張清單＝在學高中生 × 本期有修數學課；課名沒含「數學」的數學課在課程設定補標科目
  - 未定細節（動工前確認）：作業狀態欄位（提案：已交/未交＋備註）、試聽課不點名（提案）、attendance/grades/homework 的 Firestore schema（先擬進資料結構.md 給老闆過）
  - 預估：2~3 個工作段

- [ ] ⏳ **請假／補課／統計也改讀登記簿（備註名字 → studentId）**
  - 為什麼：2026-06-14 課程卡顯示已切到登記簿，但「標記請假」的勾選名單、補課容量計算、學生統計（getStudentStats）仍讀 Calendar 備註 → 同一門課「卡片人數」與「請假可勾的人」可能不一致（登記簿有、備註沒有的學生：卡片看得到、請假勾不到）
  - 子項：
    - 請假勾選名單（absence.js `buildAbsPanel`）改用 `eventRoster`（登記簿名冊），不再讀 `e.students`
    - 補課容量計算（makeup.js 用 `e.students.length` 處）改用登記簿人數
    - `getStudentStats` 出缺勤統計從「名字比對備註」改為「studentId 比對」
    - `makeupScheduled.absentStudents` 從名字字串陣列 → 連 studentId（見 [`資料結構.md`](資料結構.md) 該段註記）
  - **難點**：請假仍要寫回 Calendar 標題 `【某某請假】`（Calendar 是名字制），所以「名字 → studentId」對應躲不掉，**同名問題會在這層爆**——需靠年級標注/辨識欄位消歧，動工前先想清楚對應策略
  - 預估：2~3 個工作段；不擋學費系統，但卡片/請假落差變痛時可提前插隊

## 功能 backlog

> 使用者功能，按重要程度分類。不一定按開發路線順序做，可隨時穿插

### 優先
> 對使用者體驗影響大

- [ ] 🔴 **學年封存：一鍵匯出學年報表（死線 2026-08-31，過了就來不及）**
  - 為什麼：系統只掃描當學年（9/1～隔年 8/31）的 Calendar 事件，9/1 一到，上一學年的補課、退半堂、曠課、學生統計全部從系統消失（Calendar 原始事件還在，但要手動翻）
  - 做什麼：把當學年「已完成補課、不補課退半堂、曠課、各學生出缺勤統計」匯出成 CSV 留底
  - 注意：跨 9/1 的學費結算（例如 8 月底請假、9 月初補課）依賴這份留底
  - **時程**：建議 8 月初動工（工程量約半天）、8/31 當天在系統按最終匯出（隔天 9/1 資料就換學年）
  - **若拖到沒做完**：9/1 前手動把學生管理頁與待補課清單各學期統計截圖／抄錄留底當緊急備案
