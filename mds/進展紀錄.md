### 登入免煩：行事曆 token 改存 localStorage + 重開自動靜默續授權

老闆回報「每隔一段時間就要重新登入」。根因：Google 行事曆 token 存在 **sessionStorage**，一關瀏覽器分頁就被清掉，但 Firebase 登入狀態存 localStorage 會留著——於是重開瀏覽器後 Firebase 還記得人、行事曆 token 卻沒了，系統卡在登入頁等手動點。修法（`init.js`，紅燈先說明取得確認）：

- 全部 `gtoken` 讀寫從 `sessionStorage` 換成 `localStorage`（撐過重開瀏覽器；`oauth_state` 的 CSRF 防護仍留 sessionStorage 不動）。
- `initAPIs` 末尾新增：沒有有效 token 時，若 Firebase（localStorage）還記得登入，就自動觸發 GIS 靜默續授權（`prompt:''`），省掉手動點登入按鈕；失敗（需互動）時 callback 會 hideL 並停在登入頁當 fallback。App 模式（standalone）靜默 popup 不可用，跳過自動觸發、留給使用者手動點。
- `init.js?v=5`→`v=6` 強制重抓。

**使用者影響**：重開分頁/瀏覽器後多數情況直接進系統、不再卡登入頁；token 一樣 1 小時自動失效，白名單防線（firestore.rules）不受影響。**未 commit**。

### 揪出「每小時跳選擇帳戶」真因：多帳號 + 靜默續授權沒帶 login_hint

老闆截圖顯示週期性跳出來的是 Google「**選擇帳戶**」畫面（不是 token 過期）。真因：該 Google 帳號在瀏覽器同時登了 4 個帳號，而系統的靜默續授權 `requestAccessToken({prompt:''})` **沒指定要用哪個帳號** → 多帳號時 Google 不敢自動選，就跳選擇器。修法（`init.js`，純前端零成本）：

- 新增 `getLoginHint()`（讀 firebase currentUser.email，fallback localStorage `ghint`）與 `silentReauth()`（續授權時帶 `hint`）。
- 所有靜默續授權呼叫點（requestReauth、doSignIn Case1、scheduleTokenRefresh 定時、initAPIs 重開自動續）統一改走 `silentReauth()`，固定用同一帳號悄悄換 token。
- `redirectSignIn()`（PWA 整頁跳轉）URL 補 `login_hint`，少跳一層帳戶選擇器。
- `onSignedIn` 拿到 email 時存進 localStorage `ghint` 當 hint 來源。

**使用者影響**：桌面多帳號情況下不再每小時跳「選擇帳戶」。注意這修「桌面多帳號」這條主因；PWA／Safari ITP 跨 1 小時的限制是另一軸，要根治得另走後端 refresh token（已向老闆說明，暫緩）。

### 新增獨立工具：行事曆比對（行事曆比對.html）

需求：要快速看出兩個行事曆之間的事件差異，其中一邊是蘋果（iCloud）內建行事曆、另一邊是 Google——Google Calendar API 看不到 iCloud 行事曆，無法直接拉。

做法：新增一個獨立頁面 `行事曆比對.html`（不屬主系統、不碰 Firestore），兩種模式共用同一套比對／顯示邏輯：
- **Google 模式**：沿用主系統的 OAuth（CLIENT_ID／discovery doc，scope 縮成 `calendar.readonly`），登入後從 calendarList 下拉挑兩個行事曆 + 日期範圍，`singleEvents:true` 展開重複事件後比對。
- **.ics 模式**：上傳兩個匯出的 `.ics` 檔，純本機 FileReader 解析（自寫 RFC 5545 unfold + VEVENT parser，處理 DATE／datetime／Z／TZID），不上傳、不需登入、不需伺服器（雙擊即用）。跨蘋果／Google／Outlook 通用。給老闆的選擇是 .ics 路線——補習班課表不適合設成公開網址（路線 A 訂閱進 Google 的代價），且 .ics 完全私密。

比對核心：以「標題 + 開始時間」配對，分四類輸出——只在 A／只在 B／細節不同（結束時間・地點・說明，標出 A→B 差異）／完全相同。重複性事件（.ics 的 RRULE）比的是規則不是每筆，會標 🔁 提醒。已用 node 假資料實測四類分類與 recurring 標記皆正確。

**對使用者的影響**：多一個獨立小工具可比對任兩個行事曆的差異；主系統行為完全未動。

### 「更新」按鈕：靜默失敗 → 有回饋 + 授權過期提示

回報「↻ 更新有時候按了沒反應」。查出三個成因：(1) `refreshCurrent` 全程沒有「更新中／已更新／失敗」回饋，資料沒變時畫面重繪後一模一樣，無法分辨是更新成功還是真的沒反應；(2) loadToday/loadWeek/loadMakeup 開頭只檢查「有沒有 token 物件」、不檢查是否過期，token 過期（約 1 小時；手機 App 模式的自動刷新被刻意關掉、更容易中）後打 Calendar 被 401、被 catch 靜默吃掉，按了石沉大海也沒提示；(3) 學生頁的更新只 `renderStudents()` 本地重繪，根本不重讀 Firestore，別台裝置改的資料看不到。

修法：新增 `isTokenExpired()`（看 sessionStorage `gtoken` 的 `expires_at`）；`refreshCurrent` 改 async——token 過期就跳「授權已過期 點此授權」常駐提示、不再靜默；沒過期就 `showL('更新中…')`、三頁都先重讀 Firestore 再重抓 Calendar、結束跳「已更新」或「更新失敗：原因」。重讀雲端前若本機有尚未存檔的改動（新增 `drivePendingSave` 旗標追蹤）會先 flush，避免被雲端舊值蓋掉。

**對使用者的影響**：按「更新」一定有明確反應——成功跳「已更新」、授權過期跳「授權已過期 點此授權」（點一下就能重新授權）、失敗顯示原因；學生頁按更新現在也會抓雲端最新（含別台裝置改的學生／修課資料）。

### 手機版：學生管理頁窄螢幕橫向溢出修正

真機（390px）驗收時發現學生管理頁整頁橫向溢出、卡片右側被切。查出兩個獨立漏網點：(1) 頁首按鈕列（待補提示／升年級／價目表／課表對帳／新增學生）那層 flex 漏了 `flex-wrap`，5 顆 `nowrap` 按鈕排不下就頂出 viewport、撐出橫向卷軸，把下方卡片 grid 一起推寬；(2) 學生編輯面板內部姓名列與資料列在窄螢幕未做 RWD，姓名列按鈕、家長電話被切。修法：按鈕列補 `flex-wrap:wrap`（桌機寬度夠、不換行、零影響）；編輯面板補 RWD——姓名輸入框獨佔一行、年級＋按鈕排第二行、學校/出生年/家長電話垂直堆疊。皆只影響 ≤768px（按鈕列那條桌機也安全），純 CSS/HTML。

**對使用者的影響**：手機開學生管理頁不再整頁橫向溢出、卡片不再被切；展開學生編輯面板時儲存鈕與家長電話完整可見。

### 徹底刪除學生（硬刪除）

學生編輯面板底部新增「危險區」，提供「🗑️ 徹底刪除此學生」。與既有「變更狀態」的軟刪除（搬到歷屆、資料保留）不同，這是把學生本人 + 其所有學期的修課登記從 Firestore 永久移除、不可復原，給清測試帳號與重複建檔用。防呆：刪除前跳 modal，必須打對學生姓名才放行；只刪 Firestore 的學生與修課，不碰 Google Calendar（Calendar 備註殘留另循對帳/北極星處理）。刪除鈕放編輯面板而非卡片 hover，避免 100+ 張真學生卡誤點。

**起因**：真實資料盤點時發現舊版掃描遺毒——同一個子晴在同年級被建兩筆（一筆「（國小）子晴」名字含括號、一筆乾淨「子晴」），括號塞進名字反而讓「同名同年級」警告失效（字串不同），重複沒被擋下。現有系統只有軟刪除、清不掉這類重複與測試帳號，故補上硬刪除。新版對帳建檔已自動去括號，不再產生這類資料。

**對使用者的影響**：可在學生編輯面板永久刪除學生（打名字確認）。括號年級的正確位置是 Calendar 備註、不是名字欄；重複的子晴採「留乾淨那筆、刪括號那筆」處理（兩筆修同一課，零資料損失）。
