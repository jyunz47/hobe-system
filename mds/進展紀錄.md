## 2026-06-08

### 補課時長規則：家教一對一改為維持原時長（請假時機系統 Part A）

「請假時機系統」開發路線的第一步。原本補課時長除了練習課外一律縮短為原課的一半，但家教一對一的補課應該維持完整時長（不是半堂）。

**對使用者的影響**：為家教一對一的請假安排補課時，時段長度與原課相同（例如原課 90 分，補課就是 90 分），且時段選擇器不再顯示「縮短至原時長一半」的提示。一對二、團班（3 人以上）維持原本砍半規則不變；調課本來就維持原時長，也不受影響。

**為什麼做**：這是與使用者重新釐清整套「請假時機 → 補課時長 + 學費」規則後的第一個、零風險的小改。完整規則（A/B/C 三種請假時機、曠課、家教/團班各自的補課與學費邏輯、一對二整堂調課分支）已談定，後續 Part B（請假時機按鈕、曠課標記、學費事實記錄）、Part C（學生卡顯示加退費）再做。

**對開發的影響**：改動集中在補課時長計算一處——在原本「練習課維持原時長」的條件多加「家教一對一（type==='one'）」。同步更新功能說明的時段規則段落。

### 學生資料層強化（11 步 + 後續調整）

過去 1 週把學生 schema 從 4 欄位（id/name/grade/createdAt）擴成 10 欄位（+ school/birthYear/parentPhone/status/statusChangedAt/statusNote），為後續成績登記、繳費紀錄、學生完整檔案鋪路。

**對使用者的影響**：
- 編輯學生時多 3 個欄位可填：學校、出生年（西元）、家長電話
- 學生卡新增「📝 待補資料」chip 提醒哪些還沒補（可一鍵全部隱藏，狀態跨重整保留）
- 同名同年級學生卡新增「⚠ 同名」chip，學校填了且不重複才會消失
- 點 ✕ 不再是硬刪除，跳「變更狀態」彈窗：畢業/離開/暫停 + 選填備註
- 學生管理頁分「在學 / 歷屆」分頁，歷屆內再分畢業/離開/暫停
- 新增「📅 升年級」批次按鈕：國一~高二 +1、國三→高一、高三→畢業（國小/大學跳過）
- 掃描課表時若名字配對到歷屆生，會顯示紅色「⚠ 已畢業/離開/暫停」chip，不再給更新課程按鈕，避免畢業生死灰復燃

**為什麼做**：原本 `deleteStudent` 只清 Firestore studentList，名字還留在 Calendar 備註等地方，下次掃描又拉回來——刪了個寂寞。同名學生也無法區分。決定先強化資料層、把消歧與軟刪除路徑做出來，Calendar 端清理留到階段 3 enrollment 一起做。

**對開發的影響**：
- 學生 schema 從 4 欄到 10 欄，新欄位**不批次回填**，讀取時用 fallback（`s.status||'在學'` 等）
- `getStudentStats(name)` → `getStudentStats(studentId)`，內部仍用 name 比對 Calendar 備註（沒 id 可掛）
- 學生 id 從 `Date.now()` 改用 `_lastStudentId` 單調遞增計數器，避免同毫秒撞 id
- 新寫 `mds/資料結構.md` 記 schema 與 fallback 規則

**順手整理 todos**：原「學生刪除要完全清乾淨」主要痛點已被新方案緩解，剩下 Calendar 備註字串清理留給階段 3，從「優先」挪到「想想看」並更新說明。

## 2026-05-28

### 修登入「連環 popup」風險：合併成單一 popup

5/25 改 Firebase `signInWithPopup` 後，登入流程變成「Firebase popup → 等使用者選帳號 → Calendar OAuth popup」兩個 popup 連環。Chrome 認為 user gesture 在 `await` 後失效，第二個 popup 會被擋——昨天在正式環境踩到過。

**做法**：用 `GoogleAuthProvider().addScope('calendar')`，讓 Firebase popup 一次同時要 Firebase auth 與 Calendar OAuth token，從 `credential.accessToken` 取出後設給 `gapi.client`。Token 過期續約仍走 GIS tokenClient（背景 silent，無 popup）。

**對使用者的影響**：第一次登入只跳 1 個彈窗（Google 會把兩個權限同時列出讓你同意），不再有彈窗被擋的風險。已登入過的回訪者完全無感。

### 修正式環境 OAuth 授權網域漏設

5/25 把登入改成 Firebase `signInWithPopup` 後，`jyunz47.github.io` 沒加進 Firebase Console 的「已授權網域」清單，導致正式環境登入直接失敗（`auth/unauthorized-domain`）。今天在 Firebase Console 加上後恢復。順手記錄一個衍生問題到 todos：第二個 popup（Google Calendar OAuth）容易被瀏覽器擋（user gesture 在 await 後失效），移到優先區待解。

### Calendar API 快取（30 秒 TTL）

包一層 `cachedEventList` 取代散在 6 處的直接 `events.list` 呼叫，key 用整個請求參數序列化（保證相同 timeRange 才命中），TTL 30 秒。寫操作（patch/insert/delete）後 8 處全部加上 `invalidateEventCache()` 確保不會讀到過時資料。

**對使用者的影響**：30 秒內重複切換同一天/週/補課清單不會再打 Google Calendar API。實際感受不明顯（原本也不慢），但用久了會少打很多次 API、降低用到配額上限的風險。

**對開發的影響**：未來新增 events.list 呼叫只要用 `cachedEventList`、新增寫操作別忘了 `invalidateEventCache`，就自動享受快取。

### EventStore：抽出 findEventById 與 findMakeupScheduledById

過去散在 10 處的兩個查找模式（在多份 events 陣列裡 find by id；建臨時 Map 查補課排程）整合成兩個 helper，放在 state.js。純機械重構、零行為變化。

**對使用者的影響**：完全沒有。
**對開發的影響**：
- 未來要改「在哪些陣列裡找事件」（例如新增一個 archive 陣列）只需要動 state.js 一處
- 順手把每次都建臨時 Map 的浪費操作改成 O(1) `Map.get`，補課狀態標籤渲染微微變快
- Calendar API 快取（原本同一任務的下半段）拆成下一個項目，避免一次堆太多

### parseEv 單元測試（與順手修一個小 bug）

幫核心解析函式 `parseEv` 寫了一套 43 個測試案例的單元測試，跑在瀏覽器（`tests/index.html`），純 HTML + JS 零依賴。涵蓋：基本課程類型、教室/老師、學生名單（含科目分組）、備註行、請假/調課標題、時間、邊界情況、buildTitle。

**順手修了一個 bug**：原本「短中文備註」會被誤判為學生名（例如備註寫「請帶筆」，系統會多出一個叫「請帶筆」的學生，下次掃描課表還會自動建檔）。修法：學生名判斷規則加上「指令動詞開頭排除」（請/需/別/勿/麻煩/記得/務必/注意 開頭就不算名字）。

**對使用者的影響**：以後在 Google Calendar 備註寫短中文指示句不會再誤建學生。

**對開發的影響**：未來改解析規則，重新整理 `tests/index.html` 就能立刻看到哪些案例壞掉，不用手動點 UI 驗證。

### 調課管理功能

- 標記調課：在課程卡 modal 點「調課」，輸入原因（可空白），確認後課程標題自動標記為【調課：原因】

### 1. Token 自動刷新修復
- 修正隔一段時間後讀不到資料的問題（登入狀態看起來正常但 API 呼叫失敗）
- 根本原因：從 sessionStorage 恢復 token 時，系統誤將到期時間重算為「現在 +58 分鐘」，且 setInterval 在電腦休眠後會延遲觸發
