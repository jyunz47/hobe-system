## 2026-06-13

### 「更新」按鈕：靜默失敗 → 有回饋 + 授權過期提示

回報「↻ 更新有時候按了沒反應」。查出三個成因：(1) `refreshCurrent` 全程沒有「更新中／已更新／失敗」回饋，資料沒變時畫面重繪後一模一樣，無法分辨是更新成功還是真的沒反應；(2) loadToday/loadWeek/loadMakeup 開頭只檢查「有沒有 token 物件」、不檢查是否過期，token 過期（約 1 小時；手機 App 模式的自動刷新被刻意關掉、更容易中）後打 Calendar 被 401、被 catch 靜默吃掉，按了石沉大海也沒提示；(3) 學生頁的更新只 `renderStudents()` 本地重繪，根本不重讀 Firestore，別台裝置改的資料看不到。

修法：新增 `isTokenExpired()`（看 sessionStorage `gtoken` 的 `expires_at`）；`refreshCurrent` 改 async——token 過期就跳「授權已過期 點此授權」常駐提示、不再靜默；沒過期就 `showL('更新中…')`、三頁都先重讀 Firestore 再重抓 Calendar、結束跳「已更新」或「更新失敗：原因」。重讀雲端前若本機有尚未存檔的改動（新增 `drivePendingSave` 旗標追蹤）會先 flush，避免被雲端舊值蓋掉。

**對使用者的影響**：按「更新」一定有明確反應——成功跳「已更新」、授權過期跳「授權已過期 點此授權」（點一下就能重新授權）、失敗顯示原因；學生頁按更新現在也會抓雲端最新（含別台裝置改的學生／修課資料）。

### 手機版：學生管理頁窄螢幕橫向溢出修正

真機（390px）驗收時發現學生管理頁整頁橫向溢出、卡片右側被切。查出兩個獨立漏網點：(1) 頁首按鈕列（待補提示／升年級／價目表／課表對帳／新增學生）那層 flex 漏了 `flex-wrap`，5 顆 `nowrap` 按鈕排不下就頂出 viewport、撐出橫向卷軸，把下方卡片 grid 一起推寬；(2) 學生編輯面板內部姓名列與資料列在窄螢幕未做 RWD，姓名列按鈕、家長電話被切。修法：按鈕列補 `flex-wrap:wrap`（桌機寬度夠、不換行、零影響）；編輯面板補 RWD——姓名輸入框獨佔一行、年級＋按鈕排第二行、學校/出生年/家長電話垂直堆疊。皆只影響 ≤768px（按鈕列那條桌機也安全），純 CSS/HTML。

**對使用者的影響**：手機開學生管理頁不再整頁橫向溢出、卡片不再被切；展開學生編輯面板時儲存鈕與家長電話完整可見。

### 徹底刪除學生（硬刪除）

學生編輯面板底部新增「危險區」，提供「🗑️ 徹底刪除此學生」。與既有「變更狀態」的軟刪除（搬到歷屆、資料保留）不同，這是把學生本人 + 其所有學期的修課登記從 Firestore 永久移除、不可復原，給清測試帳號與重複建檔用。防呆：刪除前跳 modal，必須打對學生姓名才放行；只刪 Firestore 的學生與修課，不碰 Google Calendar（Calendar 備註殘留另循對帳/北極星處理）。刪除鈕放編輯面板而非卡片 hover，避免 100+ 張真學生卡誤點。

**起因**：真實資料盤點時發現舊版掃描遺毒——同一個子晴在同年級被建兩筆（一筆「（國小）子晴」名字含括號、一筆乾淨「子晴」），括號塞進名字反而讓「同名同年級」警告失效（字串不同），重複沒被擋下。現有系統只有軟刪除、清不掉這類重複與測試帳號，故補上硬刪除。新版對帳建檔已自動去括號，不再產生這類資料。

**對使用者的影響**：可在學生編輯面板永久刪除學生（打名字確認）。括號年級的正確位置是 Calendar 備註、不是名字欄；重複的子晴採「留乾淨那筆、刪括號那筆」處理（兩筆修同一課，零資料損失）。

### 掃描課表轉「課表對帳」工具

「🔍 掃描課表」改為「🧾 課表對帳」：比對行事曆備註與修課登記簿，把差異分成五桶——查無此人（可建檔並登記）、修課差異（綠＋補登／紅－退課的逐項 chip）、歷屆生出現在課表（只警示、引導先復學）、同名無法區分（引導加年級標注）、完全一致（只報人數）。每一條都等使用者點了才動資料，**不再自動把學生拉回名單、不再寫舊的 courses 欄位**；退課會明確提示將移除該筆登記（含自訂單價）。比對核心是純函式，新增 11 條單元測試。

接手上個工作段未完成的實作時，測試抓到一個真 bug：「有登記但整學期沒出現在行事曆」的學生（最該被盤點的失蹤名單）會被整個漏掉，原因是把陣列當 Set 用（`.size` 恆為 undefined）。修正後 82/82 全綠，並以假資料驗收頁截圖確認五桶呈現正確。

**對使用者的影響**：對帳取代舊「掃描→更新課程」流程，全面盤點學生在這一步完成；畢業生不會再被掃描悄悄復活，登記簿成為唯一會被寫入的修課來源。是「課程卡改讀 enrollment」與學費系統的前置。

**新功能規劃：每日營運（點名＋成績＋高中數學作業）**：與老闆對齊規格——課程手動標記「需登記成績」、今日課程分兩區（成績課在上）；每堂點名只記到/未到、缺席走既有請假流程；成績每堂每生可登多筆（標籤＋分數）；高中作業每天列「在學高中生 × 修數學課」清單。排在對帳／課程卡之後、學費系統之前，點名紀錄將成為學費「實際堂數」的事實來源。詳見 todos。

### 修課登記簿 + 課程價目表（enrollment 資料層）

建立「學生 × 課程 × 期別 × 單價」的修課登記簿，取代「掃描快照 + Calendar 備註」的雙寫，成為學生×課程歸屬的事實來源。期別帶學年（跨學年可查歷史）、收費定為每堂單價×堂數、單價採課程預設價＋個人覆蓋。登入後自動把在學學生舊課程轉入（單價留空）。新增 22 條單元測試。

**對使用者的影響**：學生編輯可直接加退課、設定每堂單價與個人優惠價；學生管理頁新增「💰 價目表」。這是學費系統的地基。

**對使用者的影響**：補課清單頁不再被整學期累積的已完成卡片拉長，預設只看到待安排與已安排兩欄；要查歷史就點統計卡或標題展開。

**為什麼**：原規則任何變更都要事前三點說明，低風險小事也得等一輪回覆。改版後確認額度集中在行為改變、schema、資安、金錢邏輯這些高風險變更，來回次數變少、每次確認變快。

## 2026-06-08

### 補課時長規則：家教一對一改為維持原時長（請假時機系統 Part A）

「請假時機系統」開發路線的第一步。原本補課時長除了練習課外一律縮短為原課的一半，但家教一對一的補課應該維持完整時長（不是半堂）。

**對使用者的影響**：為家教一對一的請假安排補課時，時段長度與原課相同（例如原課 90 分，補課就是 90 分），且時段選擇器不再顯示「縮短至原時長一半」的提示。一對二、團班（3 人以上）維持原本砍半規則不變；調課本來就維持原時長，也不受影響。

**為什麼做**：這是與使用者重新釐清整套「請假時機 → 補課時長 + 學費」規則後的第一個、零風險的小改。完整規則（A/B/C 三種請假時機、曠課、家教/團班各自的補課與學費邏輯、一對二整堂調課分支）已談定，後續 Part B（請假時機按鈕、曠課標記、學費事實記錄）、Part C（學生卡顯示加退費）再做。

**對開發的影響**：改動集中在補課時長計算一處——在原本「練習課維持原時長」的條件多加「家教一對一（type==='one'）」。同步更新功能說明的時段規則段落。

### 學生資料層強化（11 步 + 後續調整）

過去 1 週把學生 schema 從 4 欄位（id/name/grade/createdAt）擴成 10 欄位（+ school/birthYear/parentPhone/status/statusChangedAt/statusNote），為後續成績登記、繳費紀錄、學生完整檔案鋪路。

**對使用者的影響**：
- 編輯學生時多 3 個欄位可填：學校、出生年（西元）、家長電話
- 學生卡新增「📝 待補資料」chip 提醒哪些還沒補（可一鍵全部隱藏，狀態跨重整保留）
- 同名同年級學生卡新增「⚠ 同名」chip，學校填了且不重複才會消失
- 點 ✕ 不再是硬刪除，跳「變更狀態」彈窗：畢業/離開/暫停 + 選填備註
- 學生管理頁分「在學 / 歷屆」分頁，歷屆內再分畢業/離開/暫停
- 新增「📅 升年級」批次按鈕：國一~高二 +1、國三→高一、高三→畢業（國小/大學跳過）
- 掃描課表時若名字配對到歷屆生，會顯示紅色「⚠ 已畢業/離開/暫停」chip，不再給更新課程按鈕，避免畢業生死灰復燃

**為什麼做**：原本 `deleteStudent` 只清 Firestore studentList，名字還留在 Calendar 備註等地方，下次掃描又拉回來——刪了個寂寞。同名學生也無法區分。決定先強化資料層、把消歧與軟刪除路徑做出來，Calendar 端清理留到階段 3 enrollment 一起做。

**對開發的影響**：
- 學生 schema 從 4 欄到 10 欄，新欄位**不批次回填**，讀取時用 fallback（`s.status||'在學'` 等）
- `getStudentStats(name)` → `getStudentStats(studentId)`，內部仍用 name 比對 Calendar 備註（沒 id 可掛）
- 學生 id 從 `Date.now()` 改用 `_lastStudentId` 單調遞增計數器，避免同毫秒撞 id
- 新寫 `mds/資料結構.md` 記 schema 與 fallback 規則

**順手整理 todos**：原「學生刪除要完全清乾淨」主要痛點已被新方案緩解，剩下 Calendar 備註字串清理留給階段 3，從「優先」挪到「想想看」並更新說明。
